# LearnPod Home Screen — Product Review

**Date:** 2026-04-06
**Reviewer:** product-reviewer (Claude)
**Scope:** `app/page.tsx`, `app/HomeClient.tsx`, `StreakBar`, `DailyGoal`, `PodCard`, `ModuleCard`, `Onboarding`, `BottomNav`
**Context:** 66 pods / 8 modules. Home feels overwhelming.

---

## Overall Rating: 6.5 / 10 — Solid foundation, missing the "one next action" clarity

The home screen has all the right ingredients (streak, goal, review queue, pods, modules) but stacks them in an undifferentiated vertical list. The user has no single, unambiguous "what do I do right now" answer. At 66 pods, that ambiguity compounds into decision fatigue. The build is feature-rich, pedagogically aware (spaced repetition, domain affinity), and accessibility baseline is decent — but the IA treats every section as equally important, and there is no true personalization of *what* to learn beyond a domain filter.

---

## P0 — Must fix before the home screen is "complete"

### P0-1. No single "Start Here" primary action
Currently a user lands and sees: greeting, streak bar, daily goal, review queue (maybe empty state), "Today's Pods" horizontal scroll of 8, then modules. There is no hero CTA that says *"Start your next pod"* with a single tap. The featured card (i=0 in sortedPods) is the de facto primary action but it's not visually promoted above the fold on mobile after the header + streak + goal + review widgets push it down.

**Fix:** Collapse StreakBar + DailyGoal into a single compact chip row (see P1-3). Promote a true hero card: "Continue: [Pod Title]" if mid-streak, or "Review: [Oldest due pod]" if reviews > 0, or "Start: [Recommended pod]" for new users. One tap to start learning.

### P0-2. Review Queue empty-state wastes prime real estate
When `reviewPods.length === 0` the code renders a subdued italic "All caught up!" line that still occupies ~60px above the fold. For the first ~3 days of use *every* user sees this, because no reviews can be due yet.

**Fix:** Hide the review section entirely until the user has at least one completed pod whose review interval has elapsed. Replace with nothing (save vertical space) or fold into the streak/goal bar as a small "0 due" badge.

### P0-3. "Today's Pods" is not actually "today's" — it's "first 8 sorted by domain preference"
`sortedPods.slice(0, 8)` is stable across sessions until the user completes something. A returning user sees the same 8 pods every visit. The title lies to the user, and there's no sense of progression or freshness. With 66 pods this gets stale fast.

**Fix:** Implement a "Today's mix" rotation:
- 1 featured pod (hero) — pinned as the CTA target
- 2–3 domain-preferred pods the user hasn't touched
- 1–2 "stretch" pods from non-preferred domains (discovery)
- 1–2 pods from an active module in progress
- Seed the shuffle with day-of-year so it's stable within a day but rotates daily.

### P0-4. No "continue where I left off" state
If a user started a pod and bailed mid-quiz, there is no "Resume [Pod Title] — 3/5 questions" card. The store has completion tracking but no "in-progress" concept surfaced on home. This is the single biggest missed retention lever for a learning product.

**Fix:** Track `lastStartedPodSlug` + progress. Surface as the hero card when it exists.

### P0-5. Lapsed-user recovery is absent
If a user hasn't opened the app in 10 days, they see the same home as everyone else — streak shows 0, greeting says "Good Morning, Matt", pods list is unchanged. No "Welcome back — here's a gentle 2-min pod to rebuild momentum" recovery flow. The streak repair flow exists but only triggers during the at-risk window, not after a lapse.

**Fix:** If `daysSinceLastActivity > 3`, show a "Welcome back" banner with a curated easy pod (low minutes, already-familiar domain) as the hero. Suppress XP/streak pressure on return day.

## P1 — High-value improvements

### P1-1. Mid-streak users get no celebration or momentum feedback on home
Streaks only celebrate at milestone numbers (7/30/100/365). A user on day 4 gets nothing differentiating their experience from day 1. No "3 days to Bronze!" progress hint.

**Fix:** In `StreakBar`, add a subtle progress-to-next-milestone micro-label: *"3 days to Bronze 🥉"*.

### P1-2. New-user home (post-onboarding, pre-first-pod) is indistinguishable from returning-user home
After finishing onboarding, the user is routed directly into a pod (good) — but if they bail and come back, they land on the full home with streak=0, XP=0, empty review queue. No tutorial nudge, no "complete your first pod to start your streak" hero. Onboarding state is binary (`onboardingComplete`), not progressive.

**Fix:** Add a "first pod" state: if `completedPods.length === 0`, show a large hero card with the preview pod from onboarding, and hide the streak + goal bars entirely until after first completion.

### P1-3. Streak + Daily Goal are two cards where one chip row would do
Vertically, `StreakBar` (~76px) + `DailyGoal` (~68px) = ~144px consumed before any learning content. On a 667px iPhone SE viewport, the header (~80) + two widgets (~144) + section header = ~260px, leaving ~300px for actual pod content above the bottom nav. That's cramped.

**Fix:** Combine into a single horizontal "stats row": `🔥 4 · 1,240 XP · 🎯 67% today`. Tap expands to the existing detail cards. Saves ~80px above fold.

### P1-4. Active Modules section has no "next chapter" CTA
`ModuleCard` shows `done / total chapters` and a progress bar — but no direct "Continue to Chapter 4: X" link. The user must tap the module, then find the next chapter, then tap again. Two extra taps on every module resume.

**Fix:** Add "Next: [chapter title] →" beneath the progress bar, tapping goes directly to that chapter's pod.

### P1-5. No filtering/sorting on "Today's Pods"
With 66 pods and only 8 shown on home, the user has no way from home to say "show me shorter pods today" or "only Financial Models today". They must navigate to `/pods` and filter there. A quick domain-chip row beneath the section header would solve this.

### P1-6. Accessibility: horizontal pod scroll lacks keyboard affordance and scroll indicators
`.pods-scroll` appears to be a horizontal overflow container. On desktop/keyboard:
- No visible scrollbar styling hint
- No arrow keys to scroll
- No `aria-label` on the scroll region
- Focus moving to an offscreen card may not auto-scroll it into view

**Fix:** Add `role="region" aria-label="Today's pods"`, ensure `scroll-snap-align` for snap points, and add left/right chevron buttons on non-touch devices.

### P1-7. Greeting is hardcoded to "Matt"
Line 62: `Your Queue,<br />Matt.` — great for a personal app today, but the onboarding asks nothing about a name, and there's no personalization path. If this is truly single-user, that's fine; if you ever add multi-user or share the app, this is a trap.

**Fix:** Pull name from store (add to onboarding) or remove the name entirely and say "Your Queue Today".

### P1-8. Streak "at risk" pulse animation has no reduced-motion guard
`@keyframes streak-pulse` runs indefinitely. Users with `prefers-reduced-motion: reduce` will get a flashing border forever. Also flame-pulse and xp-bump animations likely have the same issue.

**Fix:** Wrap all keyframe animations in `@media (prefers-reduced-motion: no-preference)`.

### P1-9. Review Queue count badge is decorative-only to screen readers
The `<span>{reviewPods.length}</span>` badge has no `aria-label`. A screen reader hears "Review Queue 3" without context. Should be `aria-label="{n} pods due for review"`.

### P1-10. Milestone celebration modal has no keyboard close
`onClick={() => setShowMilestone(false)}` on the backdrop works for mouse/touch, but there's no `Escape` keyhandler and no focus trap. Same issue in `DailyGoal` picker and `PodCard` redo popup and `StreakBar` repair modal. Four modals, four accessibility gaps.

**Fix:** Extract a shared `<Modal>` component with focus trap, Escape handler, and `aria-modal="true"`.

## P2 — Polish / nice-to-have

- **P2-1:** `getGreeting()` uses local time but isn't memoized; re-renders on any home state change. Trivial perf but a cleaner pattern is to compute once in a ref.
- **P2-2:** Domain preference sort is stable but deterministic — no shuffle, so preferred-domain users always see the same pod #1. Consider stable-daily shuffle within the preferred set.
- **P2-3:** `fade-1`, `fade-2`, `fade-3`, `fade-4` classes imply staggered fade-in. On second+ visits this is pure delay with no information payoff; consider disabling after first session or on instant returns.
- **P2-4:** `PodCard` "Already Completed" popup is helpful but the redo flow could be proactive: offer a "Review mode" (shuffled questions, no XP) vs "Redo for XP" explicitly.
- **P2-5:** No dark-mode toggle visible — the whole app appears to be dark-only via `#13100d`. Fine for launch, but a light mode is a known engagement lever.
- **P2-6:** The avatar "M" circle in the header is non-interactive. Classic affordance waste — this is where profile, settings, notification toggle belong.
- **P2-7:** `preferredDomains` is used for sorting but never shown to the user. They can't see *why* certain pods are surfaced or edit their interests post-onboarding. Add a settings entry point behind the avatar.
